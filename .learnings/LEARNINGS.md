# Agentic Video Foundry learnings

## LRN-20260718-001: 先消除社区工具名称歧义，再决定是否安装

- 类型：供应链与技术选型
- 状态：已验证

### 现象

“motion-scene”可指向多个不相干项目；精确名称的旧 RubyMotion/SceneKit gem 与 Agent 视频生产无关，而用户更可能指 Remotion Scenes。

### 解决方案

在安装前核对原仓库、SKILL.md、许可证、运行时和维护状态。`lifeprompt-team/remotion-scenes` 是 React/TSX 场景源码库，不是 Agent Skill；Agentic Video Foundry 只把它作为按项目固定 commit 的可选资产源。

### 提升条件

该模式已写入 Agentic Video Foundry 场景路由与第三方台账；未来所有名称相近的社区插件继续执行同一来源核验。

## LRN-20260718-002: 第三方 Skill Pack 应安装最小审查子集

- 类型：供应链安全
- 状态：已验证

### 现象

HyperFrames v0.7.62 的 `media-use` Skill 包含 `curl | bash` 安装建议，与本机全局安全宪法冲突；全量安装会把不需要的高权限路径带入后续任务上下文。

### 解决方案

固定 tag 与 commit，先离线检索危险命令，只安装 Agentic Video Foundry 所需的 router、core、animation、keyframes、creative、CLI 和 motion-graphics；继续由 Foundry 管理媒体生成、凭据、来源与母带。

### 提升条件

任何第三方 Skill Pack 都先做最小能力集、危险命令和自动副作用审查，不因官方来源而跳过。

## LRN-20260718-003: Lottie 进入最终视频时需要双渲染器验收

- 类型：跨渲染器兼容性
- 状态：规范化

### 现象

Text-to-Lottie 的官方正确性基线是 Skia Skottie，但最终成片可能由 Remotion 的 `@remotion/lottie` 或 HyperFrames Lottie adapter 渲染。字体、遮罩、slots 和 SVG 特性的支持面可能不同。

### 解决方案

先在官方 Skottie 播放器检查首帧、中点和末帧，再把冻结的 JSON、字体与图片交给最终渲染器，并在最终时间轴复查代表帧和短片。

### 提升条件

该规则已成为 Agentic Video Foundry 的 Lottie 资产硬闸门。

## LRN-20260720-004: 区分 GitHub 仓库不存在与沙箱凭据不可见

- 类型：发布与环境隔离
- 状态：已验证

### 现象

用户在宿主终端完成 `gh auth login` 后，Codex 沙箱内的 `gh auth status` 仍可能报告 token 无效；与此同时，向一个尚未创建的远端地址执行 `git push` 会返回 `Repository not found`。这两个现象叠加时，容易把仓库不存在误诊为用户登录失败。

### 解决方案

先核对本地 `origin`、分支和提交，再在能访问宿主 Keychain 与网络的环境中分别执行 `gh auth status` 和 `gh repo view OWNER/REPO`。若登录有效而 `gh repo view` 明确返回仓库无法解析，先创建远端仓库，再推送并比较本地 `HEAD` 与 `origin/main` 的完整提交哈希。

### 提升条件

所有 GitHub 首次发布流程都应把“认证状态”“远端对象是否存在”“推送权限”作为三个独立检查项，不再用单次 `git push` 错误推断根因。

## LRN-20260720-005: 火山语音接入要分离凭据、服务授权与音色作用域

- 类型：音频供应商接入
- 状态：已验证

### 现象

Ark 大模型 API Key 不能替代豆包语音新版控制台的 Speech API Key；`45000010` 表示凭据无效，`45000030` 表示服务未授权或授权尚未传播。同一个音色 ID 也不一定能跨服务复用，播客音色传给音频生成接口会触发 `45001115`。

### 解决方案

适配器只从 Keychain 或环境变量读取 Speech API Key，并把凭据有效性、项目内服务授权和音色作用域作为三个独立诊断层。表现力配音先用文本提示生成试听，再以选中试听作为参考音频锁定整片声线；公共标准音色仅在所属 TTS 服务内使用。

### 提升条件

新供应商或新模型接入必须记录服务类型、模型、音色来源与参考音频哈希；出现新错误码时先补可复现诊断，再扩大自动路由。

## LRN-20260720-006: 付费音频批处理必须逐资产落盘和记账

- 类型：成本控制与可恢复生成
- 状态：已验证

### 现象

长批次中间失败时，如果只在全部完成后写 manifest，已经计费并成功返回的资产会失去追踪，重试还会造成重复扣费。部分成功响应可能没有顶层 `code`，音频字段也可能位于嵌套对象中。

### 解决方案

支持 `--asset` 单资产执行；每个成功资产立即原子更新 manifest，并以“无错误且存在 audio 或 url”识别成功响应。manifest 保存模型、路由档位、提示词哈希、参考音频哈希、时长和文件哈希，但绝不保存密钥。

### 提升条件

所有付费生成器都采用试听先行、逐项生成、增量 manifest、可重入重试和无密钥审计。

## LRN-20260720-007: 视频工具链不能假设单一 Homebrew 架构路径

- 类型：本地渲染可复现性
- 状态：已验证

### 现象

Node/Remotion 进程的 PATH 可能看不到宿主终端中的 ffmpeg；Apple Silicon 与 Intel Homebrew 又分别位于 `/opt/homebrew/bin` 和 `/usr/local/bin`。

### 解决方案

母带与审计脚本按环境 PATH、Apple Silicon Homebrew、Intel Homebrew 和系统路径依次解析 ffmpeg/ffprobe，并在找不到时输出明确诊断。

### 提升条件

所有本地视频项目统一通过能力探测定位外部二进制，不再依赖交互式 shell 的隐式 PATH。

## LRN-20260723-008: 反推外部视频智能体时应提取状态契约，而不是照抄界面与工具名

- 类型：工作流反向工程
- 状态：已验证

### 现象

截图能证明外部智能体按 `brief → storyboard → style spec → keyframes → video → QA` 推进，也能暴露每阶段的产物、确认点和失败项；但截图中的节点名或 `canvas_media_create` 只属于特定宿主，无法证明隐藏提示词、模型实现或工具在本机可用。

### 解决方案

只把可观察行为提升为供应商无关的状态契约：每阶段定义输入、输出文件、不可变量、允许改动、进入条件、退出条件和质量闸门；图像与视频生成能力通过 provider adapter 路由，不把截图中的私有工具名冒充依赖。无法从证据确认的内部实现明确标为未知。

### 提升条件

今后从截图、录屏或产品演示反推任何视频工作流，都必须同时追踪可见产物、状态转移、工具调用边界与最终成片证据，不能用界面相似替代能力验证。

## LRN-20260723-009: 强风格视频需要 Identity Block、逐镜连续性和后合成文字三重锁

- 类型：视觉一致性
- 状态：已验证

### 现象

只有“手撕纸拼贴”风格词时，生成或代码动画仍会出现角色漂移、纸片弯曲、元素瞬移、连续补间、阴影方向变化、额外英文和文字乱码。单张关键帧风格正确，也不代表跨镜连续。

### 解决方案

在 `03-style-spec` 中锁定重复角色与道具的轮廓、颜色、比例、位置、边缘和阴影；在 `02-storyboard` 中为每镜记录纸片组、进入、离场、起止状态和连续性约束；精确字幕、网址、数字与命令统一后期合成。运动只允许刚性平移、分级旋转、等比缩放、遮罩揭示和静帧替换，并用固定但不规则的 2–6 帧保持节奏表达定格感。

### 提升条件

该三重锁已成为所有强风格、跨镜头视频预设的默认结构；若关键主体跨两镜以上复用，缺少任一层都不得进入全片渲染。

## LRN-20260723-010: 语音素材的语义结尾之后仍可能藏有宽频污染尾

- 类型：音频质检
- 状态：已验证

### 现象

供应商生成的旁白在可听语句结束后可能先静音，再出现一次短促的宽频噪声。仅检查转场音效清单会误判根因；本次 37 秒“刺啦”实际来自上一段旁白素材尾部，不是该时点的 SFX。

### 解决方案

对投诉时点同时检查最终母带、分轨素材和前后场界频谱。保留原始付费资产不做破坏性改写，在混音清单中记录按帧裁点，并在裁点前做短淡出；重新渲染母带后，以相同时窗的前后频谱确认宽频突刺消失。

### 提升条件

所有旁白在入主时间轴前增加“语义结束至文件结束”的尾部扫描；任何音频裁点、淡出或增益变化都使旧母带与旧响度结果失效。

## LRN-20260723-011: 扁平纸片的开合不要依赖 CSS 3D 翻面

- 类型：跨帧渲染稳定性
- 状态：已验证

### 现象

在无稳定透视上下文的纸片元素上使用 `rotateX`，浏览器逐帧渲染可能在接近共面或背面切换时出现单帧消失、反色或闪白；静态代表帧正常并不能排除播放时闪屏。

### 解决方案

定格纸片开合改为两张确定性的 2D `clip-path` 状态，通过离散帧切换并只做二维平移/旋转。验收必须抽取切换前、切换帧和切换后的连续帧组成接触表，而不是只看单张成片截图。

### 提升条件

所有扁平剪纸、拼贴和定格风格默认禁用 CSS 3D 翻面；确需三维时必须单独建立透视、背面与跨帧回归测试。
